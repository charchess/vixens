### C1 · Double système de labels de maturité — `vixens.io/tier` vs `vixens.io/maturity`

**Fichiers :** `apps/00-infra/maturity-controller/base/scripts/sync_maturity.py`, politiques Kyverno tier-gatées (~6 policies)

**Description :**

Deux systèmes coexistent sans lien :
- **`vixens.io/tier`** : posé **manuellement** dans les manifests (ex : `"diamond"` sur ~49 fichiers)
- **`vixens.io/maturity`** : calculé **dynamiquement** par le `maturity-controller` depuis les PolicyReports Kyverno

Les politiques tier-gatées (`check-qos-class`, `check-stability`, `check-authentik-sso`, `check-velero-schedule`, `check-vulnerability-scan`, `check-restore-init`) filtrent sur `vixens.io/tier`. Le maturity-controller ne pose jamais `vixens.io/tier` — il pose `vixens.io/maturity`. **Résultat : ces politiques ne se déclenchent JAMAIS.**

**Solution proposée :**

**Unifier vers `vixens.io/maturity` (PAS vers tier)**

**Phase 1 (Immédiat - 2h) :**
1. Retirer tous les labels `vixens.io/tier` des manifests (~49 fichiers)
   ```bash
   # Script d'automatisation
   find apps/ -name "*.yaml" -exec sed -i '/vixens.io\/tier/d' {} \;
   ```

2. Mettre à jour les 6 politiques tier-gatées :
   ```yaml
   # Avant (check-velero-schedule.yaml, check-stability.yaml, etc.)
   matchLabels:
     vixens.io/tier: "diamond"
   
   # Après
   matchLabels:
     vixens.io/maturity: "diamond"
   ```

3. Vérifier que le maturity-controller continue de fonctionner :
   ```bash
   kubectl get pods -A -L vixens.io/maturity
   kubectl get policyreport -A | wc -l
   ```

**Validation :**
- ✅ 0 occurrences de `vixens.io/tier` dans apps/ (sauf documentation)
- ✅ Politiques tier-gatées apparaissent dans PolicyReports
- ✅ Maturity-controller labels restent stables

**État :** non statué

**Contreproposition/Commentaire :**

⚠️ **POURQUOI PAS tier :** Si on unifie vers `tier` (manuel), on perd le scoring automatique. Le maturity-controller devient du code mort. Il faudrait maintenir manuellement 85 applications × 4 environnements = 340 labels. Intenable.

✅ **POURQUOI maturity :** Le scoring automatique via PolicyReports est la valeur centrale du système ADR-023. Conserver `tier` aspirationnel n'apporte rien si les politiques ne s'exécutent pas.

**Dépendances :**
- **BLOQUE :** C2 (diamond aspirationnel), M5 (Emerald/Orichalcum policies), M1 (velero-schedule)
- **REQUIS AVANT :** Aucun (correction autonome)

**Risque :** MEDIUM — Test en dev requis pour valider que PolicyReports ne changent pas de manière inattendue

**Effort :** 2h (script sed + 6 politiques + tests)

### C4 · Toutes les politiques Kyverno en mode `Audit` — jamais `Enforce`

**Fichiers :** `apps/00-infra/kyverno/base/policies/*.yaml` (45 ClusterPolicy)

**Description :**

Aucune politique n'est en mode `Enforce`. Les exigences Silver (probes, resources, TLS, Infisical) ne bloquent rien en production. Le système de maturité est un **tableau de bord consultatif**, pas une **barrière qualité**.

```yaml
# 100% des politiques
validationFailureAction: Audit  # ← jamais Enforce
```

**Solution proposée :**

**Passer PROGRESSIVEMENT certaines politiques en Enforce (test en dev first)**

**Phase 1 — Évaluation (1h) :**

Identifier les 7 politiques candidates pour Enforce (Silver tier) :
1. `require-probes` — Probes (liveness, readiness, startup)
2. `require-resources` — Resource requests/limits
3. `require-tls-ingress` — TLS sur Ingress
4. `require-infisical-secret` — Infisical pour secrets
5. `check-backup` — Backup profile défini
6. `add-default-labels` — Labels Kubernetes standard
7. `sizing-v2-mutate` — Sizing labels

**Phase 2 — Test en Dev (2h) :**

```yaml
# Créer overlay dev pour Kyverno policies
mkdir -p apps/00-infra/kyverno/overlays/dev/

# Patch pour 1 politique test (require-probes)
cat > apps/00-infra/kyverno/overlays/dev/enforce-probes.yaml <<EOF
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-probes
spec:
  validationFailureAction: Enforce
EOF

# Appliquer en dev
kubectl apply -k apps/00-infra/kyverno/overlays/dev/
```

**Phase 3 — Rollout progressif (1 politique / semaine) :**
- Semaine 1 : require-probes
- Semaine 2 : require-resources
- Semaine 3 : require-tls-ingress
- Semaine 4 : require-infisical-secret
- ...

**Validation après chaque rollout :**

```bash
# Vérifier qu'aucun déploiement n'est cassé
kubectl get deployments -A | grep "0/"

# Vérifier les violations bloquées
kubectl get policyreport -A -o json | jq '.items[].results[] | select(.result == "fail" and .source == "require-probes")'
```

**État :** non statué

**Contreproposition/Commentaire :**

⚠️ **RISQUE ÉLEVÉ :** Passer en Enforce peut **casser des déploiements existants**. 198 violations `require-probes` existent actuellement. Si on enforce, ces apps ne pourront plus se déployer.

🤔 **ALTERNATIVE 1 — Enforce sur nouvelles apps uniquement :**
```yaml
# Utiliser exclude pour les apps legacy
spec:
  validationFailureAction: Enforce
  exclude:
    resources:
      namespaces:
        - media-stack  # Legacy apps sans probes
        - home-automation
```

**Contre :** Crée une dette technique — les apps legacy restent non-conformes indéfiniment.

🤔 **ALTERNATIVE 2 — Enforce APRÈS correction des violations :**
1. Corriger les 198 violations probes d'abord (voir plan d'action M5)
2. PUIS passer en Enforce
3. Garantit 0 casse

✅ **RECOMMANDATION :** Alternative 2 (correction d'abord, enforce ensuite)

**Dépendances :**
- **REQUIS AVANT :** Corriger les violations existantes (198 probes, 40+ security context, etc.)
- **BLOQUE :** Rien (peut rester en Audit indéfiniment)

**Risque :** HIGH — Peut casser déploiements si violations non corrigées

**Effort :** 4h (évaluation + test + rollout progressif) APRÈS correction des violations (~2 jours de travail)

### C5 · `check-trivy-cve` et `check-vulnerability-scan` : tier Diamond incorrect

**Fichiers :** 
- `apps/00-infra/kyverno/base/policies/check-trivy-cve.yaml`
- `apps/00-infra/kyverno/base/policies/check-vulnerability-scan.yaml`

**Description :**

ADR-023 place "Zéro CVE HIGH/CRITICAL" à **Orichalcum** (L7). Les deux politiques sont annotées **Diamond** (L6). ADR-023 contient aussi une incohérence interne : la table des bypasses cite `vixens.io/cve-accepted` → "(Diamond)" tandis que l'exigence est en Orichalcum.

**Redondance fonctionnelle :** L'une inspecte le CRD `VulnerabilityReport` (trivy-operator), l'autre interroge l'API Trivy pour les pods Diamond. Deux chemins pour vérifier la même chose.

**Solution proposée :**

**Option 1 (recommandée) — Reclassifier vers Orichalcum + fusionner :**

1. Fusionner les deux politiques en une seule `check-cve-compliance.yaml` :
   ```yaml
   apiVersion: kyverno.io/v1
   kind: ClusterPolicy
   metadata:
     name: check-cve-compliance
     annotations:
       policies.kyverno.io/category: Maturity (Orichalcum)
       policies.kyverno.io/description: "Vérifie zéro CVE HIGH/CRITICAL via VulnerabilityReport CRD"
   spec:
     validationFailureAction: Audit
     rules:
       - name: check-vulnerability-reports
         match:
           resources:
             kinds: [Deployment, StatefulSet, DaemonSet]
             selector:
               matchLabels:
                 vixens.io/maturity: "orichalcum"  # Après fix C1
         validate:
           message: "CVE HIGH/CRITICAL détectés. Utiliser bypass vixens.io/cve-accepted avec justification."
           # Logique combinée des deux politiques
   ```

2. Supprimer `check-trivy-cve.yaml` et `check-vulnerability-scan.yaml`

3. Corriger ADR-023 table des bypasses :
   ```markdown
   | vixens.io/cve-accepted | Orichalcum | CVE HIGH/CRITICAL acceptés (justification requise) |
   ```

**Validation :**
- ✅ Une seule politique CVE (pas deux redondantes)
- ✅ Tier = Orichalcum (cohérent avec ADR-023)
- ✅ Bypass label documenté correctement

**État :** non statué

**Contreproposition/Commentaire :**

🤔 **ALTERNATIVE — Garder les deux politiques séparées :**
- `check-trivy-cve` : Vérifie VulnerabilityReport CRD (toutes apps)
- `check-vulnerability-scan` : Vérifie API Trivy (apps Orichalcum seulement)

**Justification :** Deux sources de données (CRD vs API) offrent redondance. Si trivy-operator tombe, l'API reste.

**Contre :** Complexité maintenue. Deux politiques à sync. Double bruit de violations.

✅ **RECOMMANDATION :** Fusionner. Si redondance nécessaire, la faire au niveau de trivy-operator (multiple replicas, HA), pas au niveau des policies.

**Dépendances :**
- **REQUIS AVANT :** C1 (label unification) pour filtrer sur `vixens.io/maturity`
- **BLOQUE :** Aucun

**Risque :** LOW — Changement purement organisationnel

**Effort :** 1h (fusion + test + suppression)

- Usage : Applications custom, CRDs simples

| Patches complexes requis | ❌ | ❌ | ✅ |

### M8 · `app.kubernetes.io/name` absent sur 30+ applications

**Description :**

Absent sur la majorité des apps media (jellyfin, radarr, sonarr, lidarr, etc.) et plusieurs autres catégories. 

**Impact :** 
- ServiceMonitor potentiellement cassés (selector par name)
- Debug difficile (`kubectl get all -l app.kubernetes.io/name=jellyfin`)
- Non-conformité au standard Kubernetes labels

**Solution proposée :**

**Ajouter le label via mutation Kyverno (automatique, 1h) :**

Créer politique mutate `add-app-name-label.yaml` :

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: add-app-name-label
  annotations:
    policies.kyverno.io/category: Best Practices
spec:
  rules:
    - name: add-name-from-deployment
      match:
        resources:
          kinds: [Deployment, StatefulSet, DaemonSet]
      mutate:
        patchStrategicMerge:
          metadata:
            labels:
              +(app.kubernetes.io/name): "{{request.object.metadata.name}}"
          spec:
            template:
              metadata:
                labels:
                  +(app.kubernetes.io/name): "{{request.object.metadata.name}}"
```

**Validation :**

```bash
# Déployer la politique
kubectl apply -f apps/00-infra/kyverno/base/policies/add-app-name-label.yaml

# Vérifier mutation sur nouveau déploiement

# Output attendu: app.kubernetes.io/name: jellyfin

# Rollout existants pour appliquer mutation
kubectl rollout restart deployment -n media-stack jellyfin
```

**État :** non statué

**Contreproposition/Commentaire :**

🤔 **ALTERNATIVE — Ajouter manuellement dans 30+ manifests :**
